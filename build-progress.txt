## Iteration 1 — infra-001: Database Schema Setup
Date: 2026-04-07

### What was built
- Complete Drizzle ORM schema with 26 tables covering all data models from build-spec.md
- Tables: user, session, account, verification (Better Auth), workspace, member, team, teamMember, workflowState, label, project, projectTeam, projectMember, projectMilestone, cycle, issue, issueLabel, issueRelation, comment, reaction, initiative, initiativeProject, customView, notification, apiKey, webhook
- 9 PostgreSQL enums: workflowStateCategory, issuePriority, issueRelationType, projectStatus, projectPriority, initiativeStatus, memberRole, notificationType, viewLayout, estimateType
- Full Drizzle relations defined for all tables (query builder support)
- Indexes on all foreign keys and common query patterns
- Unique constraints on composite keys (team+workspace, issue+team number, etc.)

### Test results
- 39 unit tests passing (tests/schema.test.ts)
- TypeScript typecheck: clean
- Biome lint/format: clean
- Pre-existing failures in tests/onboard-validation.test.ts (10 tests, not related to this change)

### Files changed
- src/lib/db/schema.ts — complete rewrite with all tables, enums, and relations
- tests/schema.test.ts — new unit test file (39 tests)
- tests/e2e/smoke.spec.ts — initial smoke test
- biome.json — added target-docs to ignore list (pre-existing graphql lint errors)
- ralph-config.json — auto-formatted by biome

### Decisions
- Used text IDs for Better Auth tables (user, session, account, verification) to match Better Auth's convention
- Used UUID IDs for all application tables (workspace, team, issue, etc.)
- Issue identifiers stored as varchar(20) for "ENG-123" format
- Workspace settings and team settings stored as JSONB for flexibility
- Added apiKey and webhook tables beyond the base data model for API/webhook features in settings

### Infrastructure
- AWS preflight.sh running in background to provision VPC, RDS, ElastiCache, S3, ECR, ECS, ALB
- DB push will happen once RDS is available

## Iteration 2 — infra-002: Redis Connection Setup
Date: 2026-04-07

### What was built
- Redis client module (src/lib/redis.ts) with ioredis
- Two separate Redis clients: primary (caching/commands) and subscriber (pub/sub)
- Cache helpers: cacheGet, cacheSet, cacheDel, cacheDelPattern (JSON serialization, TTL support)
- Pub/sub helpers: publish, subscribe, unsubscribe
- Retry strategy with exponential backoff (max 5s)
- Lazy connect for startup resilience
